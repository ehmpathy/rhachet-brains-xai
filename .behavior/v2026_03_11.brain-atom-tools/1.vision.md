# vision: xai brain atom tool use

## the outcome world

### day-in-the-life

a developer who builds agentic workflows wants to give grok access to external tools — say, a weather lookup or a customer search. today, they handle tool calls manually with raw xai api. after this upgrade:

```ts
import { genBrainPlugToolDeclaration, asBrainPlugToolDict } from 'rhachet';

const brainAtom = genBrainAtom({ slug: 'xai/grok/4.1-fast-wout-reason' });

// define tool with typed schemas
const weatherLookup = genBrainPlugToolDeclaration({
  slug: 'weather.lookup',
  name: 'Weather Lookup',
  description: 'get current weather for a city',
  schema: {
    input: z.object({ city: z.string() }),
    output: z.object({ temp: z.number(), conditions: z.string() }),
  },
  execute: async ({ invocation }) => {
    const weather = await fetchWeather(invocation.input.city);
    return { temp: weather.temp, conditions: weather.conditions };
  },
});

// build tool dict for O(1) lookup
const tools = [weatherLookup];
const toolDict = asBrainPlugToolDict(tools);

// initial prompt
let result = await brainAtom.ask({
  prompt: 'what is the current weather in austin?',
  schema: { output: z.object({ summary: z.string() }) },
  plugs: { tools },
  role: {},
});

// loop until output is defined
while (result.calls) {
  // execute each invocation
  const executions = await Promise.all(
    result.calls.tools.map(async (invocation) => {
      const tool = toolDict[invocation.slug];
      return tool.execute({ invocation }, context);
    }),
  );

  // continue episode with tool results
  result = await brainAtom.ask({
    on: { episode: result.episode },
    prompt: executions,  // BrainPlugToolExecution[]
    schema: { output: z.object({ summary: z.string() }) },
    plugs: { tools },
    role: {},
  });
}

// result.output is now defined
console.log(result.output.summary);
```

### before/after contrast

| before | after |
|--------|-------|
| manual xai api calls for tool use | `plugs: { tools }` on atom.ask() |
| custom parse of tool_calls response | typed `result.calls.tools[]` |
| manual JSON schema conversion | zod schema auto-converted via genBrainPlugToolDeclaration |
| no structured output with tools | output schema + tools together |
| unclear cost tracked with tools | `metrics.cost` tracks tool rounds |
| manual tool result construction | `tool.execute()` auto-produces BrainPlugToolExecution |
| linear tool search | `asBrainPlugToolDict` for O(1) lookup |

### the "aha" moment

the moment the value clicks: the entire tool loop is ~15 lines. define tools with zod, ask with plugs, loop while calls, execute in parallel, continue — done. no manual xai sdk setup, no JSON schema conversion, no manual result construction.


## user experience

### usecases and goals (scoped to wish)

1. **tool invocation with output schema**: ask grok with tools plugged + structured output
2. **tool invocation without output schema**: ask grok with tools plugged, no structured output
3. **tool continuation**: execute tools, feed results back via `BrainPlugToolExecution[]`, get final answer

**out of scope for this work**:
- parallel tool call tests (xai supports by default, but not in wish)

### contract inputs & outputs

**input plugs**:
```ts
plugs: {
  tools: [
    {
      slug: string;           // "weather.lookup"
      name: string;           // "Weather Lookup" — display name
      description: string;    // helps brain decide when to use
      schema: {
        input: z.Schema;      // zod schema for tool arguments
        output: z.Schema;     // zod schema for tool result
      };
      // execute: only present for repl grain (atom callers manage execution)
    }
  ]
}
```

**prompt type** (conditional on tools):
```ts
// when tools are plugged, prompt accepts either:
prompt: string                      // initial question or follow-up
prompt: BrainPlugToolExecution[]    // tool results for continuation

// BrainPlugToolExecution shape:
{
  exid: string;      // correlation ID from invocation
  slug: string;      // tool identifier
  input: TInput;     // what tool was called with
  signal: 'success' | 'error:constraint' | 'error:malfunction';
  output: TOutput;   // tool result (or { error: Error } on failure)
  metrics: { cost: { time: IsoDuration } };
}
```

**output shape**:
```ts
{
  output: TOutput | null;  // null when brain defers to tools
  calls: {
    tools: [
      {
        exid: string;   // provider ID for correlation
        slug: string;   // which tool
        input: unknown; // parsed arguments
      }
    ]
  } | null;
  metrics: BrainOutputMetrics;
  episode: BrainEpisode;
  series: null;  // always null for atom
}
```

### usage timeline

1. **define tools** — use `genBrainPlugToolDeclaration` to create tools with type inference
2. **create lookup dict** — use `asBrainPlugToolDict(tools)` for O(1) slug lookup
3. **ask with plugs** — pass tools in `plugs.tools`
4. **loop while calls** — `while (result.calls)` until brain produces output
5. **execute invocations** — `Promise.all` to execute each tool in parallel
6. **continue episode** — call `ask()` with `prompt: executions` and `on: { episode }`
7. **use output** — `result.output` is defined after loop exits


## mental model

### how users describe this

> "i just plug in my tools and grok tells me which ones to call"

> "it's like a toolbox for grok — i define what each tool does, and grok decides when to use them"

### analogies

- **tools as function callbacks**: define the signature, model calls when needed
- **tools as API endpoints**: declare the contract, model routes requests
- **tools as plugins**: plug-and-play capabilities for the brain

### term map

| user term | rhachet term |
|-----------|--------------|
| "function" | BrainPlugToolDefinition |
| "function call" | BrainPlugToolInvocation |
| "function result" | BrainPlugToolExecution |
| "arguments" | invocation.input |
| "call ID" | invocation.exid |
| "tool result signal" | execution.signal ('success' \| 'error:constraint' \| 'error:malfunction') |
| "create tool" | genBrainPlugToolDeclaration (factory with auto type inference + error classification) |
| "tool lookup" | asBrainPlugToolDict (O(1) lookup by slug with per-slug type preservation) |


## evaluation

### how well does it solve the goals? (scoped to wish)

| goal | score | notes |
|------|-------|-------|
| tools invocable | ✅ | maps to xai function call API |
| w/ output schema | ✅ | confirmed: json_schema + tools works (grok 4 family) |
| w/o output schema | ✅ | standard tool use, no structured output |
| tool continuation | ✅ | prompt accepts BrainPlugToolExecution[] for continuation |
| type safety | ✅ | zod schemas throughout |

### pros

- **consistent contract**: same tool definitions work across providers
- **type-safe**: zod schemas for inputs, typed invocation outputs
- **progressive complexity**: no tools = no null checks on output
- **observable**: metrics track cost even with tool rounds

### cons

- **schema translation overhead**: zod → JSON schema conversion each call
- **no stream with tools**: xai returns tool calls in single chunk

### pit-of-success edgecases

| edgecase | how we handle |
|----------|---------------|
| tools plugged, brain answers directly | `calls` is null, `output` has value |
| brain requests unknown tool slug | caller validates against their tool list |
| multiple parallel tool calls | `calls.tools` is an array, process all |
| no tools plugged | `calls` is always null (type narrowed) |
| tool execution error | caller sets `signal: 'error:constraint'` or `'error:malfunction'` |

### atom vs repl tool behavior

| grain | tool behavior | tool execution |
|-------|---------------|----------------|
| atom | outputs `BrainPlugToolInvocation[]` | caller executes tools, creates `BrainPlugToolExecution[]`, continues episode |
| repl | executes tools internally via `tool.execute()` | automatic, uses `genBrainPlugToolDeclaration` wrapper |

for atoms, the caller manages tool execution. use `genBrainPlugToolDeclaration` to create tools that auto-wrap results with time and signal classification.


## open questions & assumptions

### assumptions

1. xai function call API follows openai-compatible format (confirmed via docs)
2. parallel_tool_calls enabled by default (confirmed)
3. structured output (json_schema) works alongside tools (needs validation in tests)

### questions (resolved via research)

1. **what does "w/o output schema" mean?** — RESOLVED

   the rhachet BrainAtom contract REQUIRES `schema: { output: z.Schema<TOutput> }`. it is not optional.

   so "w/o output schema" refers to the xai API parameter, not the rhachet contract:
   - **w/ output schema** = send `response_format: json_schema` to xai (provider-validated JSON)
   - **w/o output schema** = do NOT send `response_format` (free-form text, client-side parsing)

   both still require passing `schema: { output }` to satisfy the rhachet contract.

2. **test scope** — RESOLVED

   based on extant repo patterns:
   - **integration tests**: test genBrainAtom internals with real xai API
   - **acceptance tests**: test the published SDK contract (public exports only)


## what is awkward?

### output nullability with tools

when tools are plugged, `output` becomes `TOutput | null`. this is correct (brain may defer to tools), but creates null-check noise even when the caller knows the brain answered directly.

potential resolution: offer a `result.didCallTools()` type guard.

### schema translation on every call

zod → JSON schema conversion happens per invocation. for high-frequency tool use, this could be cached.

potential resolution: memoize JSON schema generation by schema hash.
