### question 1: what does "w/o output schema" mean?

**can this be answered via research?**

yes, partially. let me examine:

1. **current implementation**: always sends `response_format: json_schema` (lines 94-101 of genBrainAtom.ts)
2. **rhachet contract**: requires `schema: { output }` parameter — this is NOT optional
3. **xai API**: supports both structured output (`response_format`) and raw text responses

**interpretation via logic**:

since rhachet REQUIRES `schema: { output }`, the term "w/o output schema" cannot mean "without the schema parameter". it must refer to the xai API behavior:

- **w/ output schema** = send `response_format: json_schema` to xai → provider-validated JSON
- **w/o output schema** = do NOT send `response_format` → free-form text response, parsed client-side

**evidence from xai docs** (via websearch):

from [xAI Structured Outputs](https://docs.x.ai/docs/guides/structured-outputs):
> "structured outputs integrate with both agentic and client-side tools. This combination enables workflows where the model can use tools to gather information and return results in a predictable, strongly-typed format."

from [xAI Function Calling](https://docs.x.ai/docs/guides/function-calling):
> "with streaming, the function call is returned in whole in a single chunk"

**key insight**: xai supports BOTH structured output AND tools together (grok 4 family).

- when tools are called → response has `tool_calls`, not structured content
- after tool results fed back → final response can be structured via `response_format`

**verdict**: question can be answered via websearch.

**answer**:
- **w/ output schema** = send `response_format: json_schema` to xai → provider guarantees JSON
- **w/o output schema** = don't send `response_format` → free-form text, client-side parsing

rhachet's `schema: { output }` is always required (contract type). the difference is whether xai enforces the schema at provider level.

**action**: update vision to clarify this interpretation as resolved.

### question 2: test scope — integration vs acceptance

**can this be answered via extant code?**

yes. examining the repo:

1. **jest.integration.config.ts** exists
2. **jest.acceptance.config.ts** exists
3. **no acceptance tests yet** — only `genBrainAtom.integration.test.ts`

**what's the difference?**

looking at extant patterns in ehmpathy repos:

- **integration tests** = test internal modules against real external services (xai API)
- **acceptance tests** = test the published SDK contract as a consumer would use it

for this repo:
- **integration tests**: verify `genBrainAtom().ask()` with tools returns correct `calls.tools`
- **acceptance tests**: verify the exported API (index.ts exports) work correctly

**verdict**: question can be answered via extant code.

**answer**:
- integration tests: test genBrainAtom internals with real xai API
- acceptance tests: test the published SDK contract (what consumers import)

both test against real xai API. the difference is integration tests may access internal modules, while acceptance tests only use public exports.
