## second aspect: invocation output shape

**vision shows (line 134-141):**
```ts
calls: {
  tools: [
    {
      exid: string;
      slug: string;
```

**does "extract invocations" convey the right intent?**

- xai `id` → rhachet `exid`

- xai `function.arguments` → rhachet `input`

**potential drift:** could confuse `function.name` with tool `name` field. but the vision says `slug` is the identifier, and openai format uses `function.name` for the tool slug (not display name).

**verdict:** ADHERES — the extraction is correctly implied

## fourth aspect: output nullability semantics

**vision says (line 133):** `output: TOutput | null; // null when brain defers to tools`

**vision edgecase (line 216):** `tools plugged, brain answers directly | calls is null, output has value`

**blueprint codepath (line 43-44):**
```

if no tool_calls: parse content as before
```

**is the null semantics correct?**

the vision defines two states:
1. brain calls tools → `calls` has value, `output` is null
2. brain answers directly → `calls` is null, `output` has value

the blueprint codepath shows these as branches. the "parse content as before" path produces output.

**potential drift:** could set both null, or neither null. the branches are mutually exclusive and complete.

**verdict:** ADHERES — null semantics correctly understood

## fifth aspect: "w/o output schema" definition

**vision resolved question (line 242-248):**
```
w/ output schema = send response_format: json_schema to xai
w/o output schema = do NOT send response_format
```

**blueprint test (line 57):** `tool invocation without output schema | calls.tools populated, output null`

**is the blueprint correct about what "w/o output schema" indicates?**

this is subtle. the rhachet contract REQUIRES `schema.output` on the ask() call. so "w/o output schema" cannot refer to "omit the schema parameter."

the vision clarifies: it refers to NOT transmitting `response_format` to the xai API. both test scenarios still pass `schema.output` to satisfy rhachet. the difference is whether we ask xai to enforce JSON structure.

**potential drift:** could assume we need to make `schema.output` optional. the vision explicitly says both require the schema — it's the API parameter that differs.

**verdict:** ADHERES — blueprint tests both API variants, not contract variants

## what i specifically looked for but did not find

### misread: tool schema.output purpose

**concern:** does the blueprint use tool.schema.output correctly?

**investigation:** for atoms, the caller executes tools. the atom doesn't validate tool results — only callers do. the blueprint correctly uses tool.schema.input for API conversion and doesn't mention tool.schema.output at all. this is correct for atom grain.

### misread: exid fabrication

**concern:** does the blueprint fabricate exid instead of pulling from provider?

**investigation:** the blueprint says "extract invocations" which implies pulling values from response. xai returns `tool_calls[].id` which maps to `exid`. no fabrication.

### misread: parallel tool handler

**concern:** does the blueprint break if multiple tools are called?

**investigation:** the blueprint returns `calls.tools` as array (plural). even though parallel test is excluded per YAGNI, the array structure is preserved. single or multiple tools both work.
