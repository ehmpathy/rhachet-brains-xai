### assumption 1: BrainPlugToolDefinition maps to openai tools format

**blueprint says:** "convert tool definitions to openai tools format"

**what we assume:** BrainPlugToolDefinition fields map cleanly to openai `ChatCompletionTool` type.

**what if opposite true?** fields might not align. extra fields ignored? required fields absent?

**investigation:**

rhachet BrainPlugToolDefinition:
```ts
{
  slug: string;
  name: string;
  description: string;
  schema: { input: z.Schema; output: z.Schema };
}
```

openai ChatCompletionTool:
```ts
{
  type: 'function';
  function: {
    name: string;
    description: string;
    parameters: JSONSchema;  // from schema.input
  };
}
```

**mapping:**
- `type: 'function'` — hardcoded
- `function.name` ← `slug` (not `name`! name is display only)
- `function.description` ← `description`
- `function.parameters` ← `z.toJSONSchema(schema.input)`
- `schema.output` — NOT sent to xai (caller-side typing only)

**verdict:** assumption holds BUT slug→name mapping must be explicit. not a trivial 1:1.

// edge cases

**empty array edge case:** if `prompt: []`, what happens?

- rhachet contract: `AsBrainPromptFor<TPlugs>` types this

**verdict:** assumption holds. but should fail-fast on empty array — no valid usecase.

### assumption 3: BrainPlugToolExecution maps to openai tool message

**blueprint says:** "convert executions to openai tool messages"

**what we assume:** execution structure maps to openai's tool result format.

**investigation:**

rhachet BrainPlugToolExecution:
```ts
{
  exid: string;
  slug: string;
  input: TInput;
  signal: 'success' | 'error:constraint' | 'error:malfunction';
  output: TOutput | { error: Error };
  metrics: { cost: { time: IsoDuration } };
}
```

openai tool result message:
```ts
{
  role: 'tool';
  tool_call_id: string;  // ← exid
  content: string;       // ← JSON.stringify(output)
}
```

**mapping:**
- `role: 'tool'` — hardcoded
- `tool_call_id` ← `exid`
- `content` ← `JSON.stringify(output)`
- `slug`, `input`, `signal`, `metrics` — NOT sent (atom-side metadata)

**question:** what about error signals?

when `signal: 'error:constraint'` or `'error:malfunction'`:
- `output` contains `{ error: Error }`
- should we send error message to xai? or throw?

**decision needed:** vision shows caller constructs BrainPlugToolExecution. error handling is caller responsibility. atom just serializes output regardless of signal.

**verdict:** assumption holds. but error signal handling needs test coverage.

### assumption 9: output is null when tools called

**blueprint says:** "return `calls.tools` with invocations, `output` as null"

**what we assume:** when brain requests tool use, output should be null.

**what if opposite true?** brain might return partial output alongside tool call.

**investigation:** checked openai behavior.

when `finish_reason: 'tool_calls'`:
- `message.content` may be null or contain text
- `message.tool_calls` contains the calls

rhachet contract: `AsBrainOutputOutputFor<TPlugs>` is `TOutput | null` when tools plugged.

**question:** what if xai returns BOTH content AND tool_calls?

this can happen! brain might say "let me look that up" (content) AND request tool (tool_calls).

**decision:** if tool_calls present, set output to null. content is intermediate, not final answer.

**verdict:** assumption holds. aligns with rhachet contract semantics.
