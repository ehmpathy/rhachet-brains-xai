## why each standard holds

### rule.require.arrow-only

all three cast operations and all new ask() code use arrow functions. no `function` keyword appears. this holds because:
- cast operations are pure transforms — arrow syntax matches their stateless nature
- arrow functions bind `this` lexically, prevents accidental context bugs
- consistent with entire extant codebase (`genBrainAtom` was already arrow)

### rule.require.input-context-pattern

every procedure uses `(input: { ... })` pattern:
- `castIntoXaiToolDef({ tool })` — tool is named, not positional
- `castFromXaiToolCall({ call })` — call is named
- `castIntoXaiToolMessages({ executions })` — executions is named

this holds because named keys make call sites self-documenting. `castFromXaiToolCall({ call })` reads clearly vs a positional `castFromXaiToolCall(call)`.

### rule.require.what-why-headers

all three cast files have jsdoc with `.what`, `.why`, `.note`:

```ts
/**
 * .what = cast rhachet tool definition to xai tool format
 * .why = explicit boundary between rhachet domain and provider SDK
 * .note = xai uses openai-compatible "tools" parameter with type "function"
 */
```

this holds because headers explain purpose (not implementation), enabling future readers to understand why the code exists.

### rule.require.immutable-vars

all variables use `const`:
- `const tools` (line 81)
- `const promptIsToolExecutions` (line 97)
- `const executions` (line 100)
- `const toolCalls` (line 128)
- `const invocations` (line 129)
- `const content` (line 134)
- `const output` (line 137)
- `const calls` (line 209)

no `let` or `var` in new code. this holds because immutable bindings prevent accidental mutation bugs and make data flow explicit.

### rule.require.narrative-flow

code reads top-to-bottom with paragraph comments:
- `// convert plugs.tools to xai tools format`
- `// add prompt message (string or tool execution results)`
- `// extract tool invocations from response (if present)`
- `// extract content from response`
- `// parse output (strict mode guarantees schema conformance when content present)`
- `// build calls property`

no else branches. conditional logic uses explicit if statements (lines 98, 104). this holds because narrative flow makes code scannable — each paragraph has one purpose.

### rule.forbid.gerunds

no gerunds in names:
- `invocations` — noun (ok)
- `executions` — noun (ok)
- `promptIsToolExecutions` — noun + predicate (ok)

avoided problematic terms:
- no `existing` (would use `extant` or `found`)
- no `processing` (would use `process` or `processed`)
- no `handling` (would use `handle` or `handler`)

this holds because gerunds are ambiguous — is it a noun or verb? explicit nouns are clearer.

### rule.require.order.noun_adj

`promptIsToolExecutions` follows [noun][predicate] pattern. `toolCalls` follows [subject][noun] pattern. this holds because noun-first enables autocomplete discovery — type `prompt` to see all prompt variants.

### rule.forbid.as-cast (checked)

reviewed for `as` casts. found two in genBrainAtom.ts:
- line 213: `output as BrainOutput<TOutput, 'atom', TPlugs>['output']`
- line 214: `calls as BrainOutput<TOutput, 'atom', TPlugs>['calls']`

these are documented with comment on line 211: `// use type assertion for conditional types that TypeScript cannot infer`

this is acceptable per rule exception: "allowed only at external org code boundaries; must document via inline comment". the typescript type system cannot infer conditional types from generic parameters without assertion.
