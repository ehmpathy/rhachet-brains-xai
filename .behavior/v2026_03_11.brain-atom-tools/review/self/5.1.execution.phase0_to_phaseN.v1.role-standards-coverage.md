## absent patterns check

### rule.forbid.undefined-inputs

**question**: do any cast operation inputs use optional attributes?

**verified each cast signature**:
- `castIntoXaiToolDef(input: { tool: BrainPlugToolDefinition })` — no `?:`, all required
- `castFromXaiToolCall(input: { call: OpenAI.ChatCompletionMessageToolCall })` — no `?:`, all required
- `castIntoXaiToolMessages(input: { executions: BrainPlugToolExecution[] })` — no `?:`, all required

**answer**: no violations. all inputs are required. the `askInput.plugs?:` in genBrainAtom.ts is allowed per exception — it's part of the rhachet brain contract (user-facing), not an internal procedure.

### rule.require.shapefit

**question**: are there any `as` casts or `any` types?

**searched codebase changes**:
- cast files: zero `as` casts, zero `any` types
- genBrainAtom.ts lines 213-214: two `as` casts present

**verified the two casts**:
```ts
output: output as BrainOutput<TOutput, 'atom', TPlugs>['output'],
calls: calls as BrainOutput<TOutput, 'atom', TPlugs>['calls'],
```

these are documented on line 211: `// use type assertion for conditional types that TypeScript cannot infer`

**why these are acceptable**: the BrainOutput type uses conditional types based on TPlugs. TypeScript cannot narrow `null | T` to the correct conditional branch without assertion. the types ARE correct — the assertion helps the compiler, not hiding a bug.

**answer**: no violations. the two `as` casts are at the rhachet contract boundary and documented.

### error boundaries

**question**: should cast operations have try/catch for JSON.parse?

**answer**: no. per YAGNI review (usecase.4), error boundaries for malformed responses were explicitly excluded. the wish focuses on success path. standard parse errors provide sufficient context.

### validation

**question**: should tools parameter be validated before conversion?

**answer**: no. rhachet contract types (`BrainPlugToolDefinition`) enforce shape at compile time. runtime validation would duplicate type system.

### unit tests for casts

**question**: should cast operations have dedicated unit tests?

**answer**: no. cast operations are simple transforms (< 20 lines each). integration tests exercise them with real API calls. unit tests would duplicate coverage without added value.

### idempotency

**question**: should any new operations be idempotent?

**answer**: n/a. cast operations are pure functions — same input always produces same output. genBrainAtom.ask() is already idempotent (stateless inference).
