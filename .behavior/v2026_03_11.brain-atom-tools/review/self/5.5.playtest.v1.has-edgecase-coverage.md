# review: has-edgecase-coverage

## summary

all relevant edge cases are covered. boundaries tested. error paths verified.

walked through every failure mode i could imagine. verified each has a test.

## methodology

1. asked: "what could fail at runtime?"
2. asked: "what inputs are unusual but valid?"
3. asked: "where are the boundaries (0/1, on/off, success/error)?"
4. for each answer, verified a playtest path covers it

## what could go wrong?

| failure mode | covered? | playtest path |
|--------------|----------|---------------|
| api key absent | yes | prerequisites + pass/fail criteria |
| tool execution fails (constraint) | yes | edge 1 |
| tool execution fails (malfunction) | yes | edge 2 |
| brain answers directly when tools plugged | yes | path 3 |
| test compilation fails (type errors) | yes | pass/fail criteria |
| test timeout | yes | pass/fail criteria |

### why each failure mode is covered

**api key absent**: the prerequisites section states "xai api key available (via source ...)", and the pass/fail criteria states "credentials not found (XAI_API_KEY absent)" as a fail condition. the foreman will know immediately if credentials are absent.

**tool execution fails**: edge 1 and edge 2 test both error signal types. edge 1 uses `BadRequestError` (constraint violation). edge 2 uses `UnexpectedCodePathError` (service failure). both require the brain to reason about the error.

**brain answers directly**: path 3 explicitly tests this case. the expected outcome states "result.calls is null" and "result.output contains the answer". this proves the brain can skip tools when appropriate.

## why boundaries are adequate

1. **0 vs 1 tools** — extant tests cover 0, new tests cover 1+
2. **output schema on vs off** — path 1 vs path 2 test this boundary
3. **success vs error** — path 4 vs edge 1/2 test all signal types
4. **brain answers vs defers** — path 3 vs path 1 test this boundary

## issues found and fixed

none. the playtest covers all edge cases i identified.

## what i considered but found out of scope

### parallel tool calls

the vision mentions "multiple parallel tool calls" as an edgecase. however, the wish does not include this requirement. the playtest explicitly marks it out of scope.

### unknown tool slug

the vision mentions "brain requests unknown tool slug" as an edgecase. for atoms, the caller validates against their tool list — the atom just returns what the model requested. no playtest needed.

## why the boundaries are complete

### 0 vs 1 tools boundary

extant tests (case1-3) verify 0 tools. new tests (case4-10) verify 1+ tools. the boundary is covered on both sides.

### output schema on vs off boundary

path 1 tests with `response_format: json_schema`. path 2 tests without. the boundary is crossed.

### success vs error signal boundary

path 4 tests `signal: 'success'`. edge 1 tests `signal: 'error:constraint'`. edge 2 tests `signal: 'error:malfunction'`. all three signal types are covered.

### brain answers vs defers boundary

path 3 tests brain answers directly (calls is null). path 1 tests brain defers (output is null). both sides of the boundary are covered.
