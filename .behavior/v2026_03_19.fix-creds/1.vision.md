# 1.vision: credential supplier context for genBrainAtom

## the outcome world

### before

```ts
// credentials hardcoded to env vars
// every call reads process.env.XAI_API_KEY
// plaintext credentials floating through call stacks
// no way to inject credentials from a vault, secrets manager, or context

const atom = genBrainAtom({ slug: 'xai/grok/4' });
await atom.ask({ ... }); // reads process.env.XAI_API_KEY internally
```

problems:
- credentials leaked into env vars (security risk in shared environments)
- no way to rotate credentials mid-session
- no way to use different credentials for different contexts
- testing requires env var pollution

### after

```ts
// credentials supplied via context — getter pattern avoids plaintext
const context = genContextBrainSupplier<'xai', BrainSuppliesXai>({
  creds: async () => ({ XAI_API_KEY: await keyrack.get({ env: 'prod', key: 'XAI_API_KEY' }) }),
});

const atom = genBrainAtom({ slug: 'xai/grok/4' });
await atom.ask({ ... }, context); // creds fetched just-in-time
```

benefits:
- credentials never stored in plaintext
- credentials sourced from any secrets manager
- credentials can be rotated without restart
- credentials scoped to specific operations
- fallback to env var for simple usecases

### the "aha" moment

> "wait — i can inject credentials from my vault at call time, and they're never stored in memory as plaintext? and it just falls back to env vars when i don't care?"

## user experience

### usecase 1: simple local development

goal: quick iteration without ceremony

```ts
// just works — falls back to process.env.XAI_API_KEY
const atom = genBrainAtom({ slug: 'xai/grok/code-fast-1' });
await atom.ask({ role: {}, prompt: 'hello', schema: { output: z.object({}) } });
```

### usecase 2: production with secrets manager

goal: secure credential management in production

```ts
// context from infrastructure layer
const context = genContextBrainSupplier<'xai', BrainSuppliesXai>({
  creds: async () => ({
    XAI_API_KEY: await keyrack.get({ env: 'prod', key: 'XAI_API_KEY' }),
  }),
});

// passed through the call stack
await atom.ask({ ... }, context);
```

### usecase 3: multi-tenant credential isolation

goal: different credentials per customer

```ts
const getContextForCustomer = (input: { customerUuid: string }) =>
  genContextBrainSupplier<'xai', BrainSuppliesXai>({
    creds: async () => ({
      XAI_API_KEY: await keyrack.get({ owner: input.customerUuid, env: 'prod', key: 'XAI_API_KEY' }),
    }),
  });

// customer A uses their own xai account
await atom.ask({ ... }, getContextForCustomer({ customerUuid: 'customer-a' }));

// customer B uses their own xai account
await atom.ask({ ... }, getContextForCustomer({ customerUuid: 'customer-b' }));
```

### usecase 4: testing without env pollution

goal: isolated tests without global state

```ts
given('atom with mocked credentials', () => {
  const context: ContextBrainSupplier<'xai', BrainSuppliesXai> = {
    'brain.supplier.xai': {
      creds: async () => ({ XAI_API_KEY: 'test-key-for-this-suite' }),
    },
  };

  // tests don't pollute global env
  await atom.ask({ ... }, context);
});
```

## mental model

### how users describe this to a friend

> "instead of hardcoding api keys in env vars, you pass a function that fetches them when needed. so the key never sits in memory — it's fetched just-in-time and used immediately."

### analogies

| concept | analogy |
|---------|---------|
| getter pattern | badge reader — you tap to get access, don't carry the key |
| context injection | giving a waiter your dietary restrictions vs posting them on the wall |
| env fallback | spare key under the mat — works when you forget, but not ideal |

### terms

| user term | our term |
|-----------|----------|
| "credentials" | `creds` |
| "key supplier" | `ContextBrainSupplier` |
| "fetch credentials" | `creds()` getter |
| "fallback" | `process.env` lookup |

### type structure

```ts
ContextBrainSupplier<TSupplierSlug, TSupplierSupplies> = {
  ['brain.supplier.' + TSupplierSlug]: TSupplierSupplies
}
```

the context carries the supplies:
- `TSupplierSlug` = supplier identifier (e.g., `'xai'`)
- `TSupplierSupplies` = what you give the supplier (e.g., `BrainSuppliesXai`)

example: `ContextBrainSupplier<'xai', BrainSuppliesXai>` = `{ 'brain.supplier.xai': BrainSuppliesXai }`

**why "supplies"?** the supplier (this repo) takes in small supplies (creds) and returns bigger supplies (BrainAtoms, BrainRepls). give it a little, get a lot more back.

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| support context-injected creds | yes | via `ContextBrainSupplier<'xai', BrainSuppliesXai>` |
| avoid plaintext credentials | yes | getter returns promise, not string |
| fallback to env vars | yes | checked when context not provided |
| future-proof for rhachet lift | yes | type designed for extraction |

### pros

- **secure by design**: credentials fetched just-in-time, never stored
- **flexible sourcing**: any secrets manager, vault, or env var
- **zero breaking changes**: env var fallback preserves backwards compat
- **testable**: inject test credentials without global state
- **multi-tenant ready**: different creds per context

### cons

- **async overhead**: credential getter is async (latency depends on getter implementation — vault calls may add 50-200ms)
- **type complexity**: `ContextBrainSupplier<'xai', BrainSuppliesXai>` is a new type to learn
- **migration burden**: extant code works, but doesn't benefit until updated

### pit of success

| edgecase | how we handle it |
|----------|-----------------|
| no context, no env var | throw clear error: "XAI_API_KEY required — provide via context or env" |
| getter throws | propagate error with helpful message |
| getter returns undefined | treat as absent, fall back to env |
| creds cached vs fresh | always call getter — user controls cache |
| both `context.openai` and `context['brain.supplier.xai']` | precedence: creds getter > openai client > env var |

## what is awkward?

### the openai client construction

current code creates a new `OpenAI` client if not in context:

```ts
const openai = (context?.openai as OpenAI | undefined) ??
  new OpenAI({
    apiKey: process.env.XAI_API_KEY,
    baseURL: 'https://api.x.ai/v1',
  });
```

with a getter, we'd need to `await` before constructing:

```ts
const apiKey = context?.['brain.supplier.xai']?.creds
  ? (await context['brain.supplier.xai'].creds()).XAI_API_KEY
  : process.env.XAI_API_KEY;

const openai = new OpenAI({ apiKey, baseURL: 'https://api.x.ai/v1' });
```

this is slightly more verbose but still clear.

### namespace depth

`context['brain.supplier.xai'].creds()` is verbose. alternatives:

| option | pros | cons |
|--------|------|------|
| `context['brain.supplier.xai'].creds()` | explicit namespace, future-proof | verbose |
| `context.brain.xai.creds()` | shorter | conflicts with nested brain patterns |
| `context.xai.creds()` | shortest | namespace collision risk |

recommendation: stick with explicit namespace — it's verbose but unambiguous.

### credential key name

`XAI_API_KEY` matches env var convention but feels like shouting in code. alternatives:

| option | pros | cons |
|--------|------|------|
| `XAI_API_KEY` | matches env var | shouty |
| `apiKey` | clean | doesn't match env var |
| `xaiApiKey` | camelCase | inconsistent with env |

recommendation: use `XAI_API_KEY` for direct mapping to env var convention.

### getter called per-ask

the getter is called on every `ask()` invocation. if the getter hits a vault api, and the user calls `ask()` 1000 times, that's 1000 vault api calls.

**mitigation**: pit of success section notes "always call getter — user controls cache". but users who don't read carefully might:
- hit rate limits on their vault
- incur unexpected vault costs
- add latency to every call

**potential fix**: document clearly, or consider a "warn if getter latency > threshold" in implementation.

### cache recommendation

the getter is called fresh per `ask()`. for high-frequency usage, recommend `with-simple-cache` + `simple-in-memory-cache`:

```ts
import { createCache } from 'simple-in-memory-cache';
import { withSimpleCache } from 'with-simple-cache';

const cache = createCache();
const getXaiCreds = withSimpleCache(
  async () => ({ XAI_API_KEY: await keyrack.get({ env: 'prod', key: 'XAI_API_KEY' }) }),
  { cache },
);

const context = genContextBrainSupplier<'xai', BrainSuppliesXai>({
  creds: getXaiCreds,
});
```
